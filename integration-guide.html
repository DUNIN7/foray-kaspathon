Integration Guide

FORAY Protocol is designed to work alongside existing enterprise systems, not replace them. This guide covers the integration architecture, ERP-specific adapter patterns, and implementation strategies for connecting your systems to FORAY's tamper-evident audit infrastructure.

Why FORAY + ERP

Every major ERP claims comprehensive audit trails. They are all telling the truth — and missing the point. Native ERP audit capabilities answer: "What changed, when, and by whom?" FORAY answers: "Can you prove it to someone who doesn't trust your systems?"

This is the difference between internal record-keeping and external proof. All ERP audit systems share a fundamental limitation: the same administrators who could commit fraud can alter the audit trail that would detect it. FORAY solves this by anchoring cryptographic proof to an external, tamper-evident ledger that no internal actor can alter.

Integration Pattern

The universal FORAY integration pattern follows five steps, regardless of source system:

  1. Extract — Pull transaction data from ERP via API, webhook, or scheduled export
  2. Transform — Map to FORAY's 4-component model (Arrangements, Accruals, Anticipations, Actions)
  3. Hash — Generate component hashes, compute merkle root, apply privacy obfuscation
  4. Anchor — Submit merkle root to Kaspa blockchain
  5. Store — Archive full transaction JSON with blockchain anchor reference

Non-Disruptive Design

FORAY adapters read from your ERP — they never modify source system data. No ERP configuration changes are required for basic integration.

FORAY + Oracle Integration

Oracle integration uses the same adapter pattern. For Oracle EBS, extract via Concurrent Programs or REST APIs. For Oracle Cloud, use the BI Publisher extract or FBDI integration. The adapter maps Receivables (AR), Payables (AP), General Ledger (GL), and Order Management transactions to the FORAY 4A model.

Adapter Architecture

Every FORAY adapter follows the same three-phase architecture, regardless of source system: 

Phase 1: Extract
  - Connect to source system API (REST, SOAP, BAPI, etc.)
  - Pull transaction data with all relevant fields
  - Handle pagination for batch operations

Phase 2: Transform
  - Map source fields to FORAY 4A components
  - Apply privacy obfuscation (hash parties, round amounts)
  - Generate component hashes (SHA-256)
  - Compute merkle root

Phase 3: Anchor
  - Submit merkle root to Kaspa blockchain
  - Receive transaction ID and block confirmation
  - Store full FORAY JSON with anchor reference
  - Return confirmation to source system (optional)

Privacy vs. Auditability

Higher privacy levels reduce the information available for audit verification. Choose the minimum privacy level that meets your regulatory requirements. Standard privacy is sufficient for most use cases.
